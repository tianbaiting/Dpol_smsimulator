# dbreak elastic g4 pipeline runner

Drives the end-to-end ypol+zpol Sn112/Sn124 elastic dbreak →
g4input → g4output flow on remote spana03, preserving b_phi metadata
in both input and output ROOT.

- **Spec:** `docs/superpowers/specs/2026-04-30-dbreak-elastic-g4-pipeline-design.md`
- **Plan:** `docs/superpowers/plans/2026-04-30-dbreak-elastic-g4-pipeline.md`
- **Branch:** `feature/dbreak-elastic-pipeline`

## Architecture (revised 2026-04-30)

Two artifacts:

| File | Where it runs | What it does |
|---|---|---|
| `scripts/batch/sync_dbreak_to_spana.sh` | **Local** | rsync `data/qmdrawdata/qmdrawdata/20260413ypol/` to spana03 (only `dbreak.dat`). |
| `scripts/batch/run_dbreak_elastic_pipeline.py` | **spana03** | Python driver. 4 stages (`prebuild → prepare → gen-input → gen-output`) + `all` + `status`. All stages call subprocess directly — no inner ssh wrapping. |

Code travels via `git remote spana03` push (set up once with
`git remote add spana03 spana03:/home/tbt/workspace/Dpol_smsimulator`).
Spana03's repo has `receive.denyCurrentBranch=updateInstead` so push
syncs working tree automatically (run `git checkout -- rootlogon.C
setup.sh` before pushing — those files are auto-regenerated by
setup_spana.sh and always show dirty).

## Quick start

```bash
# (local) Push driver code + helper to spana03
cd /home/tian/workspace/dpol/smsimulator5.5
ssh spana03 'cd /home/tbt/workspace/Dpol_smsimulator && git checkout -- rootlogon.C setup.sh && rm -f setup.sh.bak.*'
git push spana03 feature/dbreak-elastic-pipeline

# (local) Push the 20260413ypol Sn data
bash scripts/batch/sync_dbreak_to_spana.sh

# (ssh spana03) Switch to feature branch, run pipeline.
# Note: must use the conda-env python (Python 3.10), NOT system /usr/bin/python3 (3.6).
ssh spana03
cd /home/tbt/workspace/Dpol_smsimulator
git checkout feature/dbreak-elastic-pipeline

# Either: foreground, watch live (will tie up ssh)
/home/tbt/.local/share/mamba/envs/anaroot-env/bin/python3 \
    scripts/batch/run_dbreak_elastic_pipeline.py all

# Or: nohup, survives ssh disconnect
nohup /home/tbt/.local/share/mamba/envs/anaroot-env/bin/python3 \
    scripts/batch/run_dbreak_elastic_pipeline.py gen-output \
    > logs/dbreak_elastic_pipeline/full_run.log 2>&1 < /dev/null &
disown
```

## Tunable parameters

All in `CONFIG` at the top of `run_dbreak_elastic_pipeline.py`. Most
common edits:
- `g4_jobs`: parallel sim_deuteron jobs on remote (default 8)
- `cut_axis_limit`: virtual-breakup cut threshold (MeV/c, default 150.0)
- `rotation_seed`: 0 for system seed, positive int for reproducibility
- `isotopes` / `gammas` / `*_directions`: scope of the run

## Smoke run (one isotope, one γ, one direction)

Edit CONFIG at the top of `run_dbreak_elastic_pipeline.py`, then
re-push and re-run:
```python
"isotopes":        ["Sn112"],
"gammas":          ["g050"],
"ypol_directions": ["ynp"],
"zpol_directions": ["znp"],
```
This produces 1 ypol + 10 zpol b-slice outputs, fast enough to
validate the wiring before scaling up.

## Re-running

Stages are largely idempotent. To force a re-run pass `--force` (only
prebuild honors it currently). For gen-output, the batch script
auto-skips outputs that look complete (any `<run_name>*.root` file
in the output dir > 100 KB). `rm` the relevant g4output subdirs to
force re-runs.

## Important quirks (learned during integration)

1. **GenInputRoot resolves file symlinks to canonical paths**: ypol
   data lives at `20260413ypol/...` (not `y_pol/phi_random/<tag>/`).
   `plan_filtered_tree` accounts for this.

2. **Conda env Python**: `/usr/bin/python3` on spana03 is 3.6 and
   doesn't support modern syntax. Always invoke
   `/home/tbt/.local/share/mamba/envs/anaroot-env/bin/python3`.

3. **Geant4 data env vars**: `setup_spana.sh` does NOT export
   `G4ENSDFSTATEDATA` etc. The driver pulls them from
   `geant4-config --datasets` and strips the `G4` prefix from each
   data dir basename (conda packaging mismatch — directories are
   `ENSDFSTATE3.0/`, not `G4ENSDFSTATE3.0/` as `geant4-config` reports).

4. **Field map file `180703-1,15T-3000.table`** (~895 MB) must be
   present at `configs/simulation/geometry/filed_map/`. The
   `sync_dbreak_to_spana.sh` script does NOT push it (one-time scp).

5. **Output filename includes Geant4 run id**: sim_deuteron writes
   `<run_name>0000.root` not `<run_name>.root`. Skip-if-exists in the
   batch script matches `<run_name>*.root`.

6. **Working-tree on spana03 always dirty**: `setup_spana.sh`
   regenerates `rootlogon.C` and `setup.sh` (yes, itself) on every
   source. Run `git checkout -- rootlogon.C setup.sh` before any
   `git push spana03`.
